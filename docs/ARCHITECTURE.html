ソフトウェアアーキテクチャ & 設計パターン

本プロジェクト（tetra-web）のシステム構成、適用されている設計パターン、および認証や機能モジュール等の具体的な実装構造を体系的に解説します。

💎

全体アーキテクチャとレイヤー構造

本システムは、Next.js (App Router) を基盤としたモダンなWebアプリケーションアーキテクチャを採用しています。 サーバーサイド実行（Server Components/Middleware/API Routes）とクライアントサイド実行（Client Components/React Context）を適切に分離し、セキュリティとパフォーマンスを最大化しています。

各レイヤーの責任境界

  • クライアント層（Client Side）: UIの状態保持、ユーザインタラクションの管理、Google Mapsオブジェクトのレンダリング、サービスワーカー（FCM）による通知受信を担います。
  • サーバー仲介層（Next.js Engine）: 認証ミドルウェアの適用、SEO最適化されたServer Componentsの初期HTML作成、およびCookieを活用した安全なセッション管理（NextAuth）を行います。
  • インフラ・外部サービス層（Cloud Services）: Cognitoによる安全なOIDC認証、Firebaseによるリアルタイムプッシュメッセージ送信、Google Maps APIによる地図タイル配信等を行います。

主要な適用設計パターン

プロジェクト全体でコードの一貫性、拡張性、保守性を向上させるために、複数の確立されたソフトウェアデザインパターンが適用されています。

機能別パッケージパターン

src/features/ 配下に、関連するコンポーネント、カスタムフック、型定義を集約する設計。各機能が疎結合になり、開発者が対象機能の変更箇所を容易に見つけ出せます。

サービスレイヤーパターン

APIクライアントやCognito, Firebaseとの通信ロジックを src/services/ に隠蔽（カプセル化）。コンポーネントはAPIの内部実装（リクエストヘッダーの設定方法など）に関与しません。

コンテキスト＆プロバイダー

src/contexts/ にてグローバル状態（プロフィール情報や受信箱状態）を一元管理し、バケツリレー（Props Drilling）によるコンポーネント連携を防止します。

カスタムフックパターン

ビュー（HTML/CSS構造）とライフサイクル/状態ロジックを分離。位置情報の追跡、FCM通知制御、カレンダーの期間計算などの状態ロジックをフックに抽出して再利用性を高めます。

認証アーキテクチャ & トークンリフレッシュ

本システムでは、セッション情報漏洩を避けるために NextAuth (v5) によるサーバー制御型セッション管理を採用しています。 AWS Cognito から返されたアクセストークンの有効期間が切れる前に、バックグラウンドで自動的にトークンをリフレッシュする独自のトークン監視・更新アルゴリズムを導入しています。

トークンリフレッシュのシーケンス

[Client App]             [NextAuth Middleware (Server)]           [Cognito IDP API]
     |                                 |                                 |
     |---- (A) APIリクエスト --------->|                                 |
     |                                 |--- (B) 有効期限チェック ---------| (ID/Access Token の exp を検証)
     |                                 |    (期限切れ or 残り10分未満)     |
     |                                 |                                 |
     |                                 |---- (C) トークン更新要求 -------->|
     |                                 |     (Refresh Token を送信)      |
     |                                 |                                 |
     |                                 |<--- (D) 新トークン発行 -----------|
     |                                 |     (AccessToken & IDToken)     |
     |                                 |                                 |
     |                                 |--- (E) セッションCookieの更新 ---|
     |<--- (F) レスポンス返却 ---------|                                 |

具体的な実装コードブロック (JWT Callback の抜粋)

src/services/auth.ts 内では、以下のコールバック関数によって暗号化JWT内に新旧のトークン状態が保持されます。 

async jwt({ token, account, user, profile }) {
  // 初回サインイン時はCognitoからのレスポンスをJWTに詰め込む
  if (account && user) {
    token.accessToken = account.access_token;
    token.idToken = account.id_token;
    token.refreshToken = account.refresh_token;
    token.accessTokenExpires = account.expires_at * 1000;
    token.userName = profile?.["cognito:username"];
    return token;
  }

  // アクセストークンが有効（残り10分以上）かつIDトークンが未期限ならそのまま返却
  if (Date.now() < token.accessTokenExpires - 600000 && !isTokenExpired(token.idToken)) {
    return token;
  }

  // 期限切れが近い、または切れている場合は Cognito に対してリフレッシュ処理を実行
  return await refreshTokens(token);
}
セキュリティ上のポイント

リフレッシュトークンおよびCognito Client Secretは、クライアントサイド（ブラウザ）のJavaScriptからは決してアクセスできないよう、NextAuthによるサーバーサイドの暗号化クッキー内に隠蔽されています。これによりXSS攻撃に対する高い耐久性を持ちます。

機能分割設計（Feature-based）の構造

本プロジェクトでは、大規模開発においても保守性が損なわれないよう、機能ごとのモジュール（Feature）単位でのコード集約（Co-location）設計を取っています。

機能モジュール（Feature）内の一般的な配置パターン

たとえば、地図を表示する機能 src/features/map/ の中には以下のように、その地図機能にのみ閉じたコンポーネントや型、ユーティリティが密に結合して配置されます。 

src/features/map/
├── components/          # 地図機能専用のコンポーネント (MapView, SearchBar等)
├── hooks/               # 位置情報追跡やピン制御を行うカスタムフック (useMapState)
├── types/               # 地図マーカーや座標用のローカル型定義
└── utils/               # 地図計算用関数 (距離計算など)

機能分割の利点

  1. 変更スコープの極小化: 地図機能に関する変更が発生した際、影響範囲や開発時の探索スコープが src/features/map/ 内に限定されます。
  2. 不要コードの削除容易性: ある機能を削除したい場合、その feature フォルダ全体を削除するだけで、プロジェクトの他のコードを破壊せずに安全にクリーンアップできます。
  3. 共通コンポーネントのクリーン化: src/components/ui/ には本当に汎用的な要素（共通ボタンやローディングスピナーなど）のみが配置され、特定のビジネスロジックに汚染されるのを防ぎます。
